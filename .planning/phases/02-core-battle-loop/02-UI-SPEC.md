After init, install required components:

```bash

```

These are the minimum shadcn components needed for Phase 02.

### Pretendard Font Setup

Add Pretendard Variable via CDN in `src/app/layout.tsx`:

```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/gh/orioncactus/pretendard@v1.3.9/dist/web/variable/pretendardvariable-dynamic-subset.min.css" />
```

Update `globals.css` `@theme inline` block:

```css
--font-sans: 'Pretendard Variable', var(--font-geist-sans), system-ui, sans-serif;
```

This gives Pretendard priority for Korean glyphs with Geist as fallback for Latin characters.

## Typography

| Role | Size | Weight | Line Height | Font Stack | Usage |
|------|------|--------|-------------|------------|-------|
| Body | 16px | 400 (regular) | 1.6 | Pretendard Variable | AI response text, question text |
| Label | 14px | 500 (medium) | 1.4 | Pretendard Variable | "Model A", "Model B" labels, status text, metadata |
| Heading | 24px | 700 (bold) | 1.3 | Pretendard Variable | Page title, result heading |
| Mono | 14px | 400 (regular) | 1.5 | Geist Mono | Win rate percentages, counters |

Notes:
- Body at 16px with 1.6 line-height optimized for Korean paragraph readability (hangul has denser vertical strokes than Latin)
- Only 2 font weights used in Phase 02: 400 (regular) for body/labels, 700 (bold) for headings. 500 (medium) is used sparingly for emphasis labels only.
- Korean text rendering: `word-break: keep-all` on all body text containers to prevent mid-syllable line breaks

## Interaction States

### State Machine: Battle Flow

```
[idle] --> user submits question --> [streaming]
[streaming] --> both responses complete --> [voting]
[voting] --> user clicks vote --> [reveal]
[reveal] --> user clicks "새 배틀" --> [idle]
[streaming] --> error occurs --> [error]
[error] --> user clicks "다시 시도" --> [idle]
```

### Per-State Visual Contract

#### 1. Idle State (initial / after "새 배틀")
- BattleInput visible with focus ring
- Response area hidden (not empty cards -- literally not rendered)
- Vote panel hidden
- Placeholder text in textarea: "한국어로 질문을 입력하세요"
- CTA button: "배틀 시작" (blue-600, full width below textarea on mobile, inline on desktop)
- Character counter: "0 / 2,000" in muted-foreground, bottom-right of textarea

#### 2. Streaming State
- BattleInput disabled (textarea grayed, button shows loading spinner)
- Two response cards appear simultaneously (both visible from the start)
- Each card shows "Model A" / "Model B" badge in muted style (no identity hints)
- Response text streams in character by character with StreamingIndicator (pulsing blue dot)
- Per BATTLE-02: responses stream independently at their natural speed (one may finish before the other -- this is intentional for speed comparison)
- Card that finishes streaming: StreamingIndicator disappears, subtle "완료" label appears
- Card still streaming: continues with pulsing indicator
- Vote buttons visible but disabled (opacity 50%, cursor-not-allowed) with tooltip: "두 응답이 모두 완료될 때까지 기다려주세요"

#### 3. Voting State (both streams complete)
- Both response cards show complete text, no streaming indicators
- Vote panel activates: "어떤 응답이 더 좋았나요?" heading
- Two buttons side by side:
  - "A 승리" (blue-600 outline variant, left)
  - "B 승리" (blue-600 outline variant, right)
- Hover: button fills with blue-600, text turns white
- Once clicked: selected button fills solid blue-600, unselected fades to muted
- Brief loading state (300ms) while vote is recorded

#### 4. Reveal State (after vote recorded)
- Response cards transform:
  - Model A card: top badge changes from "Model A" to actual model name (e.g., "Claude Haiku 4.5") with violet-600 colored badge
  - Model B card: top badge changes from "Model B" to actual model name (e.g., "GPT-4o-mini") with cyan-600 colored badge
  - Winning card gets a subtle green-600 left border (4px) and "선택됨" badge
- RevealPanel appears below cards:
  - Per-category win rates for each revealed model
  - WinRateBar: horizontal bar, model color fill, percentage in mono font
  - Format: "{모델명} 승률: {N}% ({W}승 / {T}전)" 
- "새 배틀 시작" button appears (blue-600 solid, centered)
- Previous question still visible (read-only) for context

#### 5. Error State
- Response cards show error styling (red-600 border-left 4px)
- Error message inside card: "응답을 생성하는 중 오류가 발생했습니다"
- Retry button: "다시 시도" (blue-600 outline)
- If only one model errors: that card shows error, the other shows its response. User cannot vote (both must succeed).

## Implementation Notes

### Zustand Store Shape (for planner reference)

The battle page requires a Zustand store with these state slices:
- `status`: BattleStatus (idle, streaming, voting, completed, error)
- `question`: string
- `responseA`: string (accumulated streaming text)
- `responseB`: string (accumulated streaming text)
- `streamingA`: boolean (is model A still streaming)
- `streamingB`: boolean (is model B still streaming)
- `winner`: 'a' | 'b' | null
- `revealedModelA`: string | null (actual model name after vote)
- `revealedModelB`: string | null (actual model name after vote)
- `winRates`: { modelA: { wins: number, total: number }, modelB: { wins: number, total: number } } | null

### Streaming Architecture Note

Per BATTLE-02, responses are streamed independently. The UI must:
1. Initiate both streams when battle starts
2. Display text as it arrives for each model independently
3. Track completion of each stream separately
4. Only enable voting when BOTH `streamingA` and `streamingB` are false
