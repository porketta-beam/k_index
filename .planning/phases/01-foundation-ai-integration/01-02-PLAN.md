    3. Create `src/app/api/battle/stream/route.ts` -- Streaming POST endpoint:
       ```typescript
       import { streamText } from 'ai';
       import { z } from 'zod';
       import { registry } from '@/lib/ai/registry';
       import { BATTLE_CONFIG } from '@/lib/ai/config';
       import { BUDGET_MODELS } from '@/lib/types';

       // Vercel function timeout -- 60s is sufficient for streaming AI responses
       // Per RESEARCH.md Pitfall 5: Streaming Timeout on Vercel
       export const maxDuration = 60;

       const requestSchema = z.object({
         prompt: z.string().min(1, 'Prompt is required').max(2000, 'Prompt must be under 2000 characters'),
         modelId: z.enum(BUDGET_MODELS),
       });

       export async function POST(req: Request) {
         try {
           const body = await req.json();
           const { prompt, modelId } = requestSchema.parse(body);

           const result = streamText({
             model: registry.languageModel(modelId),
             system: BATTLE_CONFIG.systemPrompt,
             prompt,
             maxOutputTokens: BATTLE_CONFIG.maxOutputTokens,
             temperature: BATTLE_CONFIG.temperature,
             onFinish: async ({ text, usage }) => {
               // Phase 1: Log token usage for cost monitoring only.
               // Database persistence (insertBattle/updateBattleResponse) is NOT wired here.
               // The full battle creation flow (random model pairing, position assignment,
               // battleId generation) does not exist until Phase 2.
               // This endpoint streams a single model response; Phase 2 orchestrates paired battles.
               console.log(`[battle/stream] ${modelId}: ${usage?.totalTokens ?? '?'} tokens, ${text.length} chars`);
             },
           });

           return result.toDataStreamResponse();
         } catch (error) {
           if (error instanceof z.ZodError) {
             return Response.json(
               { error: 'Invalid request', details: error.errors },
               { status: 400 }
             );
           }

           console.error('[battle/stream] Error:', error);
           return Response.json(
             { error: 'Internal server error' },
             { status: 500 }
           );
         }
       }
       ```

    Key implementation notes:
    - Use `toDataStreamResponse()` NOT `toTextStreamResponse()` -- data stream protocol supports metadata, error parts, tool calls for future phases (per RESEARCH.md Anti-Patterns).
    - `z.enum(BUDGET_MODELS)` validates modelId against the exact allowed model list -- prevents arbitrary model access.
    - Error handling: Zod validation errors return 400 with details. Provider errors (wrong API key, model unavailable) return 500.
    - `maxDuration = 60` exported at module level per Next.js convention (per RESEARCH.md Pitfall 5).
    - onFinish does NOT call database functions -- that wiring is Phase 2 scope.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client request -> Route Handler | Untrusted user input (prompt text, modelId) enters server |
| Route Handler -> AI provider API | Server sends validated request to external AI service |
| AI provider response -> client | Streaming AI text returned to browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | /api/battle/stream input | mitigate | Zod validation: `z.string().min(1).max(2000)` on prompt, `z.enum(BUDGET_MODELS)` on modelId. Rejects invalid input with 400. |
| T-02-02 | Tampering | Prompt injection via user question | accept | System prompt is server-side only (not user-editable in Phase 1). Inherent LLM risk -- cannot fully prevent prompt injection. Max prompt length (2000 chars) limits attack surface. |
| T-02-03 | Information Disclosure | AI API keys in client bundle | mitigate | API keys are env vars without `NEXT_PUBLIC_` prefix. Route Handler is server-side only. AI SDK reads keys from process.env server-side. |
| T-02-04 | Denial of Service | Excessive AI API calls | accept | Per D-01 and D-02: No per-user rate limiting in Phase 1. Any user can call /api/battle/stream unlimited times. Phase 4 season system provides global cap. Accepted risk for MVP. |
| T-02-05 | Denial of Service | Large prompt causing high token cost | mitigate | Prompt length capped at 2000 characters via Zod validation. maxOutputTokens capped at 2048. Combined cost per request bounded. |
| T-02-06 | Elevation of Privilege | Arbitrary model access | mitigate | `z.enum(BUDGET_MODELS)` restricts modelId to exactly 3 allowed budget models. Cannot access expensive models (GPT-4, Claude Opus, etc.) via this endpoint. |
</threat_model>
