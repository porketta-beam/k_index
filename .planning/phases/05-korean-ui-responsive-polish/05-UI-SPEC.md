# Phase 05 -- UI Design Contract

> Visual and interaction contract for Korean UI & Responsive Polish. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This phase restructures the existing battle UI into an arena.ai-inspired layout: bottom-pinned input, mobile swipe cards, updated Korean copywriting with mixed tone (D-06), and responsive design across 375px / 768px / 1280px+.
>
> **This contract OVERRIDES Phase 02 UI-SPEC for every element it redefines.** Phase 02 UI-SPEC remains the baseline for anything not explicitly changed here.

## Interaction States

### State Machine (unchanged from Phase 02)

```
[idle] --> user submits question --> [streaming]
[streaming] --> both responses complete --> [voting]
[voting] --> user clicks vote --> [reveal]
[reveal] --> user clicks "새 배틀 시작" --> [idle]
[streaming] --> error occurs --> [error]
[error] --> user clicks "다시 시도" --> [idle]
```

### Per-State Visual Contract (Phase 05 updates)

#### 1. Idle State

- **Header:** Sticky top, "K-Index" (24px bold) + "AI 블라인드 배틀" (14px muted)
- **Main area:**
  - CategorySelector: horizontally scrollable on mobile, flex-wrap on desktop
  - SystemPromptEditor: collapsible below category (unchanged behavior)
  - Empty state hero centered in remaining space:
    - Heading: "첫 번째 배틀을 시작해보세요!" (20px semibold)
    - Body: "한국어로 질문을 입력하면 두 AI가 경쟁합니다. 더 좋은 답변을 선택해주세요." (16px muted)
- **Input footer:** Sticky bottom, visible, enabled
  - Textarea: single line height on mobile (min-h-[56px]), expandable. Placeholder: "한국어로 질문을 입력하세요"
  - Submit button: "배틀 시작!" (primary variant, inline right of textarea on desktop, full width on mobile < 768px)
  - Character counter: "{n} / 2,000" (mono, muted, inside textarea bottom-right)

#### 2. Streaming State

- **Header:** Unchanged
- **Main area:**
  - CategorySelector + SystemPromptEditor: hidden (not rendered)
  - Question display: read-only box with muted background, "질문:" prefix in foreground weight
  - Response cards:
    - Desktop: two columns side by side
    - Mobile: swipe container with dot indicators. **Swipe works during streaming** (D-02). Both cards rendered even if one has no text yet.
    - Each card: "Model A" / "Model B" badge (secondary variant), progressive text (no skeleton -- D-05)
    - Streaming indicator: pulsing blue dot at end of text (unchanged)
    - Completed card: "완료" label + duration in mono
  - VotePanel: visible but disabled. Buttons at 50% opacity, cursor-not-allowed.
    - Heading: "어떤 응답이 더 좋았나요?" (20px semibold)
    - Wait message: "두 응답이 모두 완료될 때까지 기다려주세요" (14px muted)
- **Input footer:** Sticky bottom, visible, disabled (textarea grayed, button shows spinner + "응답 생성 중...")

#### 3. Voting State (both streams complete)

- **Header:** Unchanged
- **Main area:**
  - Question display: visible (read-only)
  - Response cards: both complete, no streaming indicators
  - VotePanel: enabled
    - "A가 더 좋아" (outline variant, min-w-[140px], min-h-[44px])
    - "B가 더 좋아" (outline variant, min-w-[140px], min-h-[44px])
    - Hover: button fills with primary color, text turns white
    - After click: selected fills solid primary, unselected fades to opacity 50%
- **Input footer:** hidden (not rendered in voting state)

#### 4. Reveal State

- **Header:** Unchanged
- **Main area:**
  - Question display: visible (read-only)
  - Response cards transform (instant, no animation -- D-04):
    - Model A badge: actual model name in violet-600 badge
    - Model B badge: actual model name in cyan-600 badge
    - Winner card: green-600 left border (4px) + "선택됨" badge
  - RevealPanel (instant appearance, no animation -- D-04):
    - Category win rates per model (WinRateBar, unchanged)
    - "새 배틀 시작" button (primary variant, centered)
- **Input footer:** hidden (not rendered in reveal state)

#### 5. Error State

- **Header:** Unchanged
- **Main area:**
  - Question display: visible if question was submitted
  - Error message: "오류가 발생했습니다" (destructive color, formal tone -- D-06)
  - Retry button: "다시 시도" (outline variant, primary color)
  - If response cards were partially rendered, they remain visible with error border
- **Input footer:** hidden (not rendered in error state)

## Mobile Swipe Specification

### CSS Scroll-Snap Implementation

Container (mobile only, hidden on md+):
```
flex snap-x snap-mandatory overflow-x-auto scrollbar-hide
```

Each card wrapper:
```
w-full flex-shrink-0 snap-start snap-always px-4
```

### scrollbar-hide Utility (add to globals.css)

```css
@layer utilities {
  .scrollbar-hide {
    scrollbar-width: none;
    -ms-overflow-style: none;
    &::-webkit-scrollbar {
      display: none;
    }
  }
}
```

### Active Card Detection

Use `IntersectionObserver` with `{ root: scrollContainer, threshold: 0.5 }`. When a card intersects at >50%, update `mobileActiveCard` in Zustand store.

### Dot Indicator Specification

- Two circles, 8px x 8px, `rounded-full`
- Active: `bg-foreground` (solid dark)
- Inactive: `bg-muted-foreground/30` (faint gray)
- Gap between dots: 8px (`gap-2`)
- Margin top from swipe container: 12px (`mt-3`)
- Centered horizontally (`flex justify-center`)
- Color transition: 150ms ease-out

### Scroll-Snap Stability During Streaming

- Each mobile card wrapper has `min-h-[300px]` to prevent layout shift during streaming
- Both cards are always rendered (even if response text is empty) to keep container dimensions stable
- Text appends inside a fixed-dimension card, only the internal scroll changes

## Response Card Internal Scroll Specification

Per D-10: when response text exceeds card height, the card scrolls internally.

| Breakpoint | Max Height | Property |
|------------|-----------|----------|
| Mobile (< 768px) | 50vh | `max-h-[50vh]` on CardContent |
| Desktop (>= 768px) | 60vh | `md:max-h-[60vh]` on CardContent |

Additional:
- `overflow-y-auto` on CardContent
- `min-h-[300px]` on the card wrapper (mobile only) for scroll-snap stability
- Scrollbar is visible inside cards (not hidden) to indicate scrollable content

## Sticky Bottom Input Specification

Per D-12: input moves from inline position to sticky bottom footer.

### Layout Rules

- Position: `sticky bottom-0` inside flex column (NOT `fixed` -- avoids iOS Safari keyboard bug)

### BattleInput State Preservation

Current `BattleInput` uses local `useState` for the question text. Moving it to a sticky footer changes its tree position, which would cause React to unmount/remount and lose typed text.

**Solution:** Add `inputText` field to Zustand store. `BattleInput` reads from and writes to the store instead of local state. The `reset()` action clears `inputText` to `""`.

### Textarea Sizing in Footer

- Desktop: `min-h-[56px]` (compact, ~2 visible lines), max grows to `max-h-[120px]`
- Mobile: `min-h-[44px]` (single visible line), max grows to `max-h-[100px]`
- Submit button: inline right of textarea on desktop (`flex flex-row`), full width below on mobile (`flex flex-col`)
