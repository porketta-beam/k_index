# Phase 04 -- UI Design Contract

> Visual and interaction contract for the Season System & Global State. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> This phase adds season lifecycle awareness to the existing battle flow. When a season ends (auto or admin), the battle UI is entirely replaced by a season-ended screen. During an active season, the battle flow is unchanged. No battle counter is displayed to users (D-04). Admin controls are API-only with no UI (D-02).

## Interaction States

### State Machine: Season + Battle Flow

```
[idle]
  |-- user submits question --> POST /api/battle/start
  |     |-- 200 OK --> [streaming] (existing battle flow, unchanged)
  |     |-- 503 season_ended --> [season-ended]
  |
[season-ended]
  |-- no user actions available (informational only)
  |-- page refresh --> POST /api/battle/start again --> still 503 --> [season-ended]
  |-- admin starts new season --> POST /api/battle/start --> 200 OK --> [streaming]

[streaming / voting / reveal / error]
  |-- all states: identical to Phase 02-03, no changes
  |-- D-07: in-flight battles complete normally even if season ends mid-battle
```

### Per-State Visual Contract

#### 1. Season-Ended State (NEW)

- **Trigger:** `/api/battle/start` returns HTTP 503 with `{ error: "season_ended", seasonNumber: N, message: "..." }`
- **Visual:** Entire battle flow (CategorySelector, SystemPromptEditor, BattleInput, ResponseCards, VotePanel, RevealPanel, empty state) is hidden. Only the SeasonEnded card is rendered.
- **Season number visible:** "시즌 {N}" in large Display typography
- **No interactive elements:** No buttons, no links, no forms. The user cannot do anything.
- **No polling:** The page does not auto-check for a new season. The user discovers a new season has started when they manually refresh and try to start a battle.
- **Header unchanged:** The "K-Index" + "AI 블라인드 배틀" header remains visible above the season-ended card.

#### 2. Battle Start Rejection Handling (Modified Idle State)

When the user submits a question and the server returns a 503 season_ended response:

1. The `handleStartBattle` function in `BattleArena` checks `res.status === 503`
2. Parses the JSON response for `error === "season_ended"`
3. Calls `store.setSeasonEnded(data.seasonNumber)` (or `store.setSeasonEnded(null)` if no seasonNumber)
4. The component re-renders, detecting `store.seasonEnded === true`, and renders `<SeasonEnded>` instead of the battle flow
5. No toast notification -- the full-screen takeover is sufficient feedback
6. No error state -- this is not an error, it is an expected end-of-season state

#### 3. In-Flight Battle During Season End (D-07)

If a season ends while a user is mid-battle (streaming, voting, or reveal phase):

- The current battle completes normally through voting and reveal
- The battle store does NOT switch to season-ended during an in-flight battle
- Only when the user clicks "새 배틀 시작" and the next `/api/battle/start` is rejected will they see the season-ended screen
- No mid-battle interruption, no toast about season ending

#### 4. All Other States (Unchanged)

Idle, streaming, voting, reveal, and error states are identical to Phase 02-03. No visual changes.

## Zustand Store Additions

Phase 04 extends the existing battle store with season state:

```typescript
// New state fields
seasonEnded: boolean;           // true when season is ended (blocks battle UI)
seasonNumber: number | null;    // season number for display (null if no season)

// New actions
setSeasonEnded: (seasonNumber: number | null) => void;  // Sets season-ended state
```

The `setSeasonEnded` action sets `seasonEnded: true` and stores the season number for display. The `reset` action does NOT clear `seasonEnded` -- once the season is over, it stays over until the user refreshes and the next `/api/battle/start` succeeds.

To clear `seasonEnded`, the user refreshes the page (Zustand store resets to initial state with `seasonEnded: false`). On the next battle start attempt, if the season is still ended, the server rejects again and the state is set back to true.

## API Response Contract

### POST `/api/battle/start` -- Season-Ended Rejection

```typescript
// HTTP 503 Service Unavailable
{
  error: "season_ended",
  seasonNumber: number | null,  // null if no active season exists
  message: string               // Korean message from server (for logging, not displayed -- UI uses its own copy)
}
```

The client checks `res.status === 503` and `data.error === "season_ended"` to distinguish from other 503 errors (e.g., server overload). If the status is 503 but the error is not `season_ended`, the existing error handling path applies.

### POST `/api/battle/vote` -- Counter Increment (No UI Changes)

The vote route now increments the Redis counter and potentially auto-ends the season. This is entirely server-side with no response changes. The vote response format is unchanged.

## Implementation Notes

### Minimal UI Surface

Phase 04's user-facing UI change is deliberately minimal. The context decision (D-04) explicitly hides the battle counter from users. The admin interface is API-only (D-02). The only visible change a user can observe is:

1. When a season ends, attempting to start a new battle shows the SeasonEnded screen
2. When a new season starts, the SeasonEnded screen disappears on next page load + battle attempt

This "surprise ending" is intentional per the context specifics: "사용자에게 남은 배틀 수를 노출하지 않음. 시즌이 갑자기 끝나는 느낌을 의도적으로 유지."

### SeasonEnded Component Structure

```tsx
interface SeasonEndedProps {
  seasonNumber: number | null;
}

// Renders a centered Card with season info
// If seasonNumber is null, omits the "시즌 N" display
// No interactive elements
```

### BattleArena Modification

The modification to `BattleArena` is a single conditional at the top of the render function:

```tsx
// Early return: season-ended state takes over the entire UI
if (store.seasonEnded) {
  return (
    <div className="w-full max-w-[1120px] mx-auto px-4 py-8">
      <header className="text-center space-y-1">
        <h1 className="text-2xl font-bold">K-Index</h1>
        <p className="text-sm text-muted-foreground">AI 블라인드 배틀</p>
      </header>
      <SeasonEnded seasonNumber={store.seasonNumber} />
    </div>
  );
}
```

The header is duplicated (not extracted) because Phase 05 (UI Polish) will refactor the layout. Premature abstraction is avoided per project conventions.

### handleStartBattle Modification

```typescript
// In handleStartBattle, before the existing !res.ok check:
if (res.status === 503) {
  const data = await res.json().catch(() => ({}));
  if (data.error === "season_ended") {
    store.setSeasonEnded(data.seasonNumber ?? null);
    return;  // Do not fall through to error handling
  }
}
```

### Win Rates Now Season-Scoped

The reveal panel's win rate display is now scoped to the current season (CONTEXT.md D-09). No visual change to the RevealPanel component -- the data source changes server-side. The win rate format remains: "{model} 승률: {N}% ({W}승 / {T}전)". At the start of a new season, all win rates show 0% (0승 / 0전) until battles accumulate.
