<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> /api/battle/start | Untrusted category ID and system prompt text from client |
| Client -> /api/battle/stream | Token-authenticated, no direct client input for category/prompt |
| Client -> /api/battle/vote | Token-authenticated, no direct client input for category |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | /api/battle/start | mitigate | Validate category ID against known CATEGORIES array using Zod .refine(); reject unknown IDs with 400 |
| T-03-02 | Tampering | /api/battle/stream | mitigate | System prompt comes ONLY from signed HMAC token (session.sp), never from client request body. Existing pattern from Phase 2 (D-08) extended to include prompt. |
| T-03-03 | Tampering | /api/battle/vote | mitigate | Category comes ONLY from signed HMAC token (session.cat), never from client request body. Prevents category spoofing in win rate queries. |
| T-03-04 | Denial of Service | /api/battle/start | mitigate | System prompt validated max 500 chars server-side via Zod .max(500). Prevents oversized prompts causing excessive token costs. |
| T-03-05 | Tampering | System prompt | accept | User-provided system prompt is user-facing (they control their own AI context). Prompt injection risk is accepted because: no privilege escalation (all budget-tier models), no access to system secrets, and 500-char limit reduces attack surface. |
| T-03-06 | Information Disclosure | HMAC token | accept | Token contains system prompt text in base64url (not encrypted). Accepted because: prompt is user-provided content, not a secret. Token is transmitted over HTTPS in POST body. |
</threat_model>
